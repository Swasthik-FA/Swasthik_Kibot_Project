# release/

Fabrication release packs for this project, organized by revision. Each revision gets its own subdirectory. A release pack is the authoritative snapshot of what was sent to the fab house for a specific revision — files here should not be modified after a release is tagged.

See SOP §11.8 for the full release pack specification. See `../FAQ.md` Q3.4 for generating release packs and Q3.5 for validating them.

## Directory layout

```
release/
├── rev_a1/       First prototype fabrication of Rev A
│   ├── gerbers/
│   ├── drill/
│   ├── bom.csv
│   ├── centroid.csv
│   ├── schematic.pdf
│   ├── fab_drawing.pdf        <- manual, per SOP §11.8
│   ├── assembly_drawing.pdf   <- manual
│   ├── release_notes.md       <- manual
│   ├── residual_risks.md      <- manual
│   ├── emc_disclosure.md      <- manual, Validated/Assured only
│   └── (Assured tier additions: release_manifest.json, SHA256SUMS.txt, production_readiness_signoff.md)
│
├── rev_a2/       Prototype respin of Rev A (ECO applied)
│   └── ...
│
├── rev_b/        First fabrication of Rev B (new revision letter)
│   └── ...
│
└── rev_b_prod/   Production release of Rev B
    └── ...
```

## Revision naming convention

- `rev_a` — initial Rev A release
- `rev_a1` — first prototype fabrication of Rev A
- `rev_a2` — second prototype spin of Rev A (after an ECO)
- `rev_b` — first fabrication of Rev B (revision bump from Rev A)
- `rev_b_prod` — production release of Rev B (after prototype validation)

Letters are sequential (A -> B -> C -> ...), skipping I, O, Q to avoid confusion with 1, 0. The revision letter MUST also appear on the silkscreen of the fabricated board (per SOP §7.4).

## What goes in a release pack (by tier)

### All tiers (the quality floor)

Scripted (generated by `scripts/export_release_pack.sh`):
- `gerbers/` — all copper layers, solder mask, silkscreen, paste, outline
- `drill/` — Excellon drill files (plated and non-plated separated)
- `bom.csv` — Bill of materials
- `centroid.csv` — Pick-and-place (X, Y, rotation, layer per component)
- `schematic.pdf` — Schematic for reference

Manual (engineer must add before release):
- `fab_drawing.pdf` — stackup, material, surface finish, tolerances, impedance targets, fab notes
- `assembly_drawing.pdf` — reference designators, pin 1 markers, DNP, orientation
- `release_notes.md` — what's in this revision, changes from previous, known issues
- `residual_risks.md` — explicit risk disclosure (things we couldn't fix or didn't test)

### Validated and Assured tiers (additional)

- `emc_disclosure.md` — EMC review scope, residual risks, client responsibility statement (per SOP §9.5)

### Assured tier only (additional)

- `release_manifest.json` — machine-readable manifest with version, tag, reviewer names, gate signoff dates, residual risks
- `SHA256SUMS.txt` — checksums for all production files
- `production_readiness_signoff.md` — Tech Lead + Project Manager formal signoff

## How to create a release

1. Make sure the design is at a commit you want to release (all gates passed, all checklists signed off)
2. Run `./scripts/export_release_pack.sh rev_<letter>[<digit>]`
3. Add the manual artifacts (fab drawing, assembly drawing, release notes, residual risks, EMC disclosure if applicable)
4. For Assured tier, add the release manifest, checksums, and production readiness signoff
5. Run `./scripts/check_release_pack.sh release/rev_<letter>[<digit>]` to verify completeness
6. Commit the release pack to git
7. Open a PR with `Stage: release`
8. After CI passes and reviewers approve, merge to `main`
9. Tag the release: `git tag -a hw-v<major.minor.patch>-rev<letter>[<digit>] -m "Release message"`
10. Push the tag: `git push origin hw-v<major.minor.patch>-rev<letter>[<digit>]`
11. Send the files from `release/rev_<letter>/` to the fab house

## Things to NOT do

- **Do NOT modify files in a release pack after it's been tagged.** Release packs are immutable snapshots. If you need a change, create a new release pack in a new subdirectory.
- **Do NOT send fab the working Gerbers from `design/pcb/outputs/`.** Always use the files in `release/rev_<X>/` — those are the authoritative release files generated by the release script.
- **Do NOT delete old release packs.** They are audit trail for every fabricated revision. Git history alone isn't enough — having the release pack files present makes re-fabrication from an old revision straightforward.
- **Do NOT commit intermediate Gerbers under `release/`.** Only actual release packs go here.
